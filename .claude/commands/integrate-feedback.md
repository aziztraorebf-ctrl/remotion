## Context
- Aziz recoit regulierement des feedbacks de sources externes :
  - Jury AI (GPT-4o, Grok, Gemini, Kimi)
  - Reviews humaines (brutales ou constructives)
  - Analyses d'autres LLM (Gemini, ChatGPT)
- Le pattern A3 est prouve : review constructive + review brutale -> synthese
- Les corrections doivent etre tracees et documentees

## Your Task

Recevoir un feedback externe, l'analyser objectivement, et integrer les points valides dans le projet.

### Step 1: Recevoir le feedback

Demander a Aziz :
- Coller le feedback complet (texte brut)
- Source : qui a donne ce feedback ? (GPT-4o, Gemini, Kimi, humain, autre)
- Cible : sur quoi porte le feedback ? (script, video, plan, style, audio)
- Contexte : quand et comment ce feedback a ete obtenu ?

### Step 2: Lecture et comprehension

Avant toute analyse, lire le document cible du feedback :
- Si script : lire la version actuelle
- Si video : lire le plan de production + voir les frames si disponibles
- Si plan : lire le plan actuel

**IMPORTANT** : Ne pas reagir au feedback avant d'avoir lu ce sur quoi il porte.

### Step 3: Analyse structuree (pattern A3)

Produire un tableau de triage :

```markdown
## Analyse du feedback de [Source] sur [Cible]

### TRIAGE

| # | Point souleve | Verdict | Action | Raison |
|---|---|---|---|---|
| 1 | [resume du point] | ACTION | [modification precise] | [pourquoi c'est valide] |
| 2 | [resume du point] | CONSIDERATION | [noter pour plus tard] | [interessant mais pas urgent] |
| 3 | [resume du point] | REJETE | - | [pourquoi ca ne s'applique pas] |
| 4 | [resume du point] | DEJA FAIT | - | [ou c'est deja couvert] |
```

**Regles de verdict :**
- **ACTION** : Le point est factuel, specifique, et ameliore objectivement le resultat
- **CONSIDERATION** : Le point est pertinent mais subjectif ou pas prioritaire
- **REJETE** : Le point est hors-scope, subjectif sans fondement, ou contredit une decision validee
- **DEJA FAIT** : Le point existe deja dans le projet (citer ou)

### Step 4: Synthese si multiples sources

Si Aziz fournit 2+ feedbacks (pattern A3) :

```markdown
## Synthese croisee

### Convergences (2+ sources d'accord)
- [Point] -> ACTION (consensus fort)

### Divergences
- [Point] : Source A dit X, Source B dit Y
  -> Decision proposee : [Z] parce que [raison]

### Points uniques pertinents
- De [Source A] : [point unique mais valide]
```

**Regle de consensus** (Jury AI) :
- 4/4 ou 3/4 d'accord -> ACTION immediate
- 2/4 d'accord -> CONSIDERATION (Aziz decide)
- 1/4 seulement -> documenter, Aziz tranche

### Step 5: Integration

Pour chaque point ACTION :
1. Identifier le fichier exact a modifier
2. Faire la modification
3. Commenter inline si le changement n'est pas evident

Documenter toutes les modifications :
```markdown
## Modifications appliquees

| # | Fichier | Ligne | Avant | Apres | Source |
|---|---|---|---|---|---|
| 1 | script-v3.md | 42 | "stat WID" | [supprime] | Consensus 4/4 |
```

### Step 6: Mise a jour memoire

- Si le feedback revele un pattern reutilisable : ajouter dans `memory/key-learnings.md`
- Si le feedback change la direction du projet : mettre a jour `memory/current-project.md`
- Si le feedback concerne un outil/API : mettre a jour `memory/apis-and-tools.md`

### Guardrails

- **NE PAS tout accepter aveulement**. Chaque point merite une analyse critique.
- **Respecter les decisions deja validees**. Si Aziz et le Jury ont deja tranche, un nouveau feedback ne peut pas annuler ca sans discussion.
- **Differencier opinion vs fait**. "Je n'aime pas le ton" = opinion. "La stat est fausse" = fait.
- **Source importes** : un feedback de Gemini sur la precision historique pese plus qu'un feedback de GPT-4o sur la precision historique (Gemini = rigueur dans notre Jury)
- **Anti-boucle** : Si le meme point revient 3+ fois sans resolution, escalader a Aziz avec les options.
- **Toujours montrer le triage a Aziz AVANT d'integrer** pour validation.
